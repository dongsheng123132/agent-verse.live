# 最简实现：打赏 + 红包（无合约）

不写智能合约，先让用户装 CFX 钱包，打赏 = 给地址打款；红包 = 后端代发。

---

## 1. 打赏（最简单）

**流程**

1. 页面上固定一个「活动收款地址」（Conflux eSpace 地址，你们自己的钱包地址）。
2. 用户自己装好 Conflux 钱包（如 Fluent），复制我们的收款地址，在钱包里选「转账」→ 填地址、金额（CFX 或 USDT 等）→ 发送。
3. 可选：页面上提供「复制收款地址」按钮、二维码（地址转成 QR），或「用 Fluent 打开」链接（若钱包支持 `conflux://` 或类似，可预填收款地址和金额，用户点一下跳转钱包）。

**不需要**

- 不写合约。
- 不一定要「连接钱包」：用户只要知道收款地址，用任何支持 Conflux 的钱包打款都行。

**可选增强**

- 前端用 Conflux 区块浏览器 API 或 RPC 查该收款地址的最近入账，展示「最近打赏」列表（展示金额、时间、tx 链接）。

---

## 2. 红包（无合约的两种做法）

红包的本质是「有人先出钱，多人来领」。不写合约的话，发钱必须有一个「中心」来执行转账。

### 做法 A：人工发（最省开发）

- **发红包**：主办方往一个指定地址打款（和打赏一样，打到你们控制的钱包）。
- **领红包**：用户在页面点「领取红包」→ 填自己的 Conflux 地址（或之后接「连接钱包」自动带出地址）→ 提交。运营/开发从钱包里手动给这些地址各转一笔（每人一份），先到先得、发完为止。
- 优点：零开发合约、零后端逻辑。缺点：要人工操作，不适合量大。

### 做法 B：后端代发（半自动，仍无合约）

- **发红包**：主办方把「红包总金额」打到你们的一个收款地址（同上，你们控制的钱包）。
- **领红包**：用户点「领取红包」→ 连接钱包或填写地址 → 后端记录「该地址已领过、本场剩余份数」，若可领则从该钱包自动转一笔 CFX/稳定币给该地址（每人每场限领 1 次，份数发完即止）。
- 需要：一台后端服务 + 一个「热钱包」（保管私钥或用托管），后端调 Conflux RPC 发起转账；数据库或缓存记录「某场次、某地址是否已领、已领多少份」。
- 优点：体验接近「点击即领」，仍不需要写链上合约。缺点：需要后端和热钱包，有托管和运维责任。

---

## 3. 总结

| 功能 | 最简做法 | 需要的东西 |
|------|----------|------------|
| **打赏** | 页面展示收款地址 + 二维码/复制按钮，用户自己用 CFX 钱包打款 | 一个 Conflux 收款地址；可选：链上查询展示最近打赏 |
| **红包** | 方案 A：人工按提交的地址手动打款；方案 B：后端 + 热钱包自动按地址转账，每人限领一次 | A：人工；B：后端服务、热钱包、Conflux RPC、简单 DB/缓存 |

先让大家下载/使用 CFX 钱包，打赏就用「给这个地址打款」即可；红包若量不大可先人工，量大了再做后端代发。
